## Core Principles

- Treat capability building as a design problem too.
- Convert vague growth goals into repeatable exercises with clear feedback.
- Build strong foundations before chasing tool breadth.
- Maintain at least one toolchain you can use fluently enough to turn an idea into a runnable experiment quickly.
- Use code, tests, and metrics to make design discussion precise.
- Start from a simple correct baseline, then iterate with measurement.
- Estimate a theory limit or practical ceiling so optimization has a stopping rule.
- Use exercises to expose unknown unknowns and missing foundations, not only to prove current skill.
- Practice with peers when possible so different solutions can be compared on shared checks.

## Derived Workflow

### Preparation

- Pick one target capability or weakness.
- Reduce the problem until it is small enough to rerun cheaply.
- Keep the essential tradeoff intact.

### Practice Loop

- Define correctness first.
- Define measurements before tuning.
- Build the baseline implementation.
- Change one thing at a time and observe the metrics.

### Reflection

- Ask which missing theory blocked progress.
- Convert that gap into the next practice topic.

## Article-Specific Warnings

- Do not confuse tool familiarity with deep design ability.
- Do not wait for production work alone to provide enough practice.
- Do not keep tuning endlessly once the likely ceiling is clear.
